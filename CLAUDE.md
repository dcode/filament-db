# Filament DB

Next.js 16 + Electron desktop app for managing 3D printing filament profiles with NFC tag support.

## Commands

```bash
npm run dev              # Next.js dev server (localhost:3456)
npm run build            # Production build
npm run lint             # ESLint
npm test                 # Vitest (single run)
npm run test:watch       # Vitest (watch mode)
npm run test:coverage    # Vitest with coverage
npm run electron:dev     # Electron + Next.js dev
npm run electron:build   # Full Electron build pipeline
```

## Architecture

- **Frontend**: Next.js App Router (TypeScript, React 19, Tailwind CSS)
- **Backend**: Next.js API routes under `src/app/api/`
- **Desktop**: Electron with esbuild-compiled main/preload (`electron/`)
- **Database**: Mongoose ODM with MongoDB Atlas (cloud), embedded MongoDB (offline), or hybrid mode
- **Tests**: Vitest with mongodb-memory-server; coverage enforced on `src/lib/` and `src/models/`
- **NFC**: @pokusew/pcsclite (native module, requires PC/SC headers on Linux; imported directly in `electron/nfc-service.ts`). Reads OpenPrintTag (NFC-V/ISO 15693) and Bambu Lab MIFARE Classic (ISO 14443-3A) tags

## Key Conventions

- **Path alias**: `@/` maps to `src/`
- **tsconfig excludes**: `node_modules`, `electron/`, `scripts/`, `packages/` (scripts use native modules unavailable in CI; `packages/` is the standalone Expo mobile app with its own toolchain)
- **Standalone output**: Next.js builds in standalone mode for Electron bundling
- **Dark mode**: Use `dark:` Tailwind variants on all UI — app supports both light and dark themes. Native controls (the `<select>` option popup, scrollbars, date/color pickers) follow the CSS `color-scheme` property, set to `dark` on `html.dark` in `src/app/globals.css` (GH #598). Without it the OS draws those controls light even in dark mode — e.g. a dropdown's option list renders white with light text. `dark:` classes only style the closed control, not the OS-drawn popup; `color-scheme` covers the rest.
- **Electron config**: electron-store for desktop persistence (connection mode, AI keys, currency, `exposeToLan`); localStorage fallback in web mode
- **IPC pattern**: `ipcMain.handle()` in `electron/main.ts`, exposed via `contextBridge` in `electron/preload.ts`, typed in `src/types/electron.d.ts`
- **Share on local network (v1.45)**: a Settings toggle backed by the electron-store key `exposeToLan` (default `false`). When off, the embedded Next server binds `localhost` (loopback-only, prior behaviour). When on, `electron/main.ts` respawns the server with `HOSTNAME=0.0.0.0` so other devices on the LAN (the mobile scanner) can reach it; the `get-lan-ip` IPC handler surfaces the reachable URL for the user. Flipping the toggle records `exposeToLanChanged` and respawns the server so the bind switches without an app restart. Pairs with mDNS for discovery. **The optional `FILAMENTDB_API_KEY` bearer gate (`src/lib/apiAuth.ts`) is NOT a usable mitigation for the packaged desktop app (#828):** it's an all-or-nothing gate with no same-origin exemption (deliberately — `Sec-Fetch`/`Origin` are spoofable by non-browser clients, #767), so setting it locks the desktop's own first-party web UI out of `/api` too. It's meant for headless / Docker deployments the mobile app or curl talk to, and there is no in-app setter (the server reads it only from its OS environment). The LAN-share Settings card discloses this posture with "there is no password".
- **mDNS auto-discovery (v1.47)**: `electron/mdns-service.ts` advertises the desktop as `_filamentdb._tcp` via `bonjour-service` (pure JS, no native dep) so the mobile app finds it without typing an IP. Advertised ONLY while `exposeToLan` is on — `syncMdnsAdvertisement()` in `electron/main.ts` starts/stops it on toggle and re-publishes after a server restart/health recovery. The mDNS instance name is suffixed with the hostname (`Filament DB (host)`) because bare-name collisions across two desktops would suppress the second on probe (#723).
- **OpenPrintTag**: CBOR encoder in `src/lib/openprinttag.ts`, NDEF wrapping in `electron/ndef.ts`. CBOR aux_region_offset must point to valid CBOR within the NDEF payload (Prusa app requirement)
- **Bambu NFC**: MIFARE Classic decoder in `src/lib/bambuTag.ts` (re-exported by `electron/bambu-tag.ts`). HKDF-SHA256 key derivation from UID, binary block parser, maps to `DecodedOpenPrintTag`. Read-only (RSA-2048 signed tags)

## Internationalization (i18n)

- **Framework**: Custom React Context-based i18n (no external library), following the useCurrency pattern
- **Provider**: `src/i18n/TranslationProvider.tsx` — provides `t(key, params?)`, `locale`, `setLocale`
- **Locales**: `src/i18n/locales/en.json` (English), `src/i18n/locales/de.json` (German) — ~1490 flat key-value pairs (count drifts on every PR — `jq 'keys|length' src/i18n/locales/en.json` for current). Since v1.34.1 the `tests/i18n-key-coverage.test.ts` invariant enforces (a) every literal `t("…")` in `src/` resolves in both locales and (b) en+de carry the same key set — a new translation key MUST land in both files in the same PR.
- **Interpolation**: `{paramName}` tokens in translation strings, e.g. `t("sync.time.minutesAgo", { count: 5 })`
- **Fallback chain**: current locale → English → raw key
- **Persistence**: electron-store (desktop) or localStorage (web), key `filamentdb-locale`
- **Settings**: Language selector in Settings page (same toggle-button pattern as Currency)
- **Adding a language**: Create `src/i18n/locales/xx.json` with all keys, add entry to `LOCALES` array in `src/i18n/index.ts`

## Testing

- ~3000 tests across ~145 files (unit + Mongoose model + Next.js route handlers). Exact counts drift on every PR — run `npm test` for the current numbers.
- **Test discovery is scoped to `tests/`** (`test.include: ["tests/**/*.test.ts"]` in `vitest.config.ts`) so a nested git worktree under `.claude/worktrees/` — or any other in-tree checkout — can't get its stale test copies swept into the run.
- Coverage thresholds: 99% lines, 98% statements, 96% functions, 96% branches (raised from 80/80/90/75 in the v1.61 coverage sweep, which drove `src/lib/**` + `src/models/**` to ~99% lines / ~98% branches). Enforced on `src/lib/**` and `src/models/**`; `src/lib/compressImage.ts` and `src/lib/labelBitmap.ts` are excluded because their main flow is DOM/canvas-only — GH #526. The residual uncovered branches are provably-unreachable defensive guards (Mongoose always materialises array schema fields as `[]` + rejects invalid Dates at cast; WHATWG URL mandates a host for http(s); regexes capturing only finite numerics; internal-consistency asserts) — thresholds sit just below the achieved numbers so a real regression trips CI while normal churn doesn't.
- **Full DB-backed suite under `--coverage` needs capped parallelism.** v8 instrumentation + ~15 concurrent `mongodb-memory-server` instances (one `mongod` per test file via `tests/setup.ts` `beforeAll`) overwhelms the machine → `MongoNetworkTimeoutError` / hook timeouts. Run coverage with `--maxWorkers=4` (or fewer); the plain `npm test` run without instrumentation is fine at full parallelism.
- Setup file: `tests/setup.ts` (mongodb-memory-server). **Caveat**: setup wipes `mongoose.models` between tests; route-level tests that use `.populate(...)` must re-register models in `beforeEach` by calling `mongoose.model(name, schema)` directly (see `tests/locations-route.test.ts` for the pattern).
- Tests run in CI on Node 20 and 22

## CI/CD

- **The release/Docker publish paths run the FULL CI gate via a reusable workflow (`ci-gate.yml`, #852)**: lint, root `tsc --noEmit`, Electron typecheck (`electron:typecheck`), coverage-enforced tests (`test:coverage`), `npm audit --audit-level=moderate`, `npm run build`, and the standalone production smoke test. `on: workflow_call`, invoked via `uses: ./.github/workflows/ci-gate.yml` by `release.yml` (a `gate` job that runs ONCE before the platform matrix; build legs `needs: gate`) and `docker.yml` (`needs: gate` before build+push). Before #852 those tag-triggered workflows re-implemented only a partial gate (lint + `npm test`), so a `v*` tag could publish from a commit branch-CI would reject (e.g. an Electron type error, since `electron/` is excluded from the root tsconfig). **Reusable-workflow refs resolve at the tagged commit**, so the tag must contain `ci-gate.yml` (true for any tag cut after #852). **`test.yml` deliberately stays INLINE** (not a caller of `ci-gate.yml`) — a reusable call would rename its checks to `gate / test (20)` etc. and break `main`'s branch protection (required contexts `test (20)`/`test (22)`/`smoke`). So `test.yml` and `ci-gate.yml` are TWO copies of the same gate that must be **kept in sync** (a cross-reference comment in each says so).
- **Tests**: `test.yml` runs the gate (test matrix Node 20+22 + smoke) inline on push to main and PRs.
- **Releases**: Triggered by `v*` tags (`release.yml`). Runs `ci-gate.yml` once, then builds macOS (x64+arm64), Windows (x64), Linux (x64+arm64). Uploads assets to GitHub release.
- **Docker**: Triggered by `v*` tags (`docker.yml`, also `workflow_dispatch`). Runs `ci-gate.yml`, then builds the multi-arch (amd64+arm64) image (native per-arch runners → digest merge, #845) and pushes to GHCR.
- **Bundled technical-reference content is committed-snapshot-only (#853)**: `release.yml` NO LONGER runs `scripts/fetch-reference.mjs` per platform leg (it used to, best-effort, so legs could ship different `src/content/` content + `REFERENCE_WIKI_SHA` for the same tag). Every artifact (desktop + Docker) now ships the committed `src/content/referenceContent.ts` + `referenceIndex.ts`. **Refresh it via `npm run fetch:reference` in a PR before tagging**; the scheduled `reference-drift.yml` (weekly + `workflow_dispatch`) re-runs the generator against the live wiki and FAILS when the committed snapshot has drifted — a visible nudge, NOT a PR gate.
- **Version bump**: `release-bump.yml` (manual `workflow_dispatch` with a semver input) bumps the version across `package.json`, `package-lock.json`, and `public/openapi.json`, opens a bump PR, and enables auto-merge — so the version-bump commit goes through the same PR + CI gate as everything else (it needs a `RELEASE_BUMP_TOKEN` PAT secret). See "Release Process".
- **Mobile (EAS) build**: `eas-build-mobile.yml` (manual `workflow_dispatch`, platform + profile inputs) queues an Expo EAS Build of `packages/mobile` — the mobile app is managed Expo (CNG), so `ios/`/`android/` aren't committed and Xcode Cloud can't build it; EAS generates the native project + signs on Expo's servers. Needs an `EXPO_TOKEN` secret; `--no-wait` so it never blocks a runner. A repo-root `.easignore` scopes the upload to `packages/mobile`. See the "Mobile scanner app" section.
- **Lint rule**: `react-hooks/set-state-in-effect` — don't call setState directly in useEffect body; use lazy initializers or callbacks

## Release Process

1. Update version in **three** places (the release CI uses package-lock for `npm ci`, OpenAPI reports the API version in `/api-docs`):
   - `package.json` (top-level `version`)
   - `package-lock.json` (two occurrences: top-level `version` and `packages[""].version`)
   - `public/openapi.json` (`info.version`)
   - **Shortcut**: `gh workflow run release-bump.yml -f version=<x.y.z>` does all three + opens an auto-merging bump PR (avoids a direct push to protected `main`). Steps 2–3 below are the manual equivalent.
2. Commit and push to main with a `Bump version to <x.y.z>` message (or let the bump PR merge)
   - **Reference content (#853):** the bundled technical reference ships from the committed `src/content/` snapshot — it is NOT re-fetched at build time. If the wiki changed since the last refresh (the scheduled `reference-drift.yml` flags this), run `npm run fetch:reference` and land the regenerated `src/content/` in a PR BEFORE tagging, so the release ships current content.
3. After the bump lands on main, pull, tag with `git tag v<version>`, and push the tag (this triggers `ci-gate.yml` → the build matrix; a commit that fails the gate now blocks the release instead of shipping, #852)
4. CI builds desktop installers and uploads to a draft GitHub release. The release is auto-published when uploads finish; **assets ≠ source of truth for "release done"** — wait for the workflow run, not just the release-created event
5. Apply release notes with `gh release edit v<version> --notes-file …`. CI doesn't overwrite the body when it publishes, so a single edit (any time after the draft is created) suffices

### Release-process gotchas
- **CSP fixes affect TWO files.** The web CSP lives in `next.config.ts`; the Electron renderer applies its OWN CSP from `electron/main.ts`'s `onHeadersReceived` handler and the renderer doesn't merge them — the value set in Electron REPLACES whatever Next sent. Any `script-src` / `frame-src` / new-directive change has to land in both. Past regressions: v1.25.0 shipped with the web CSP gated for dev `'unsafe-eval'` but the Electron CSP unchanged (v1.25.1 patch); v1.30.3 fixed an `img-src 'https:'` asymmetry from #371. The one intentional asymmetry is `connect-src`: Electron adds `ws://localhost:* http://localhost:*` for the embedded Next server; everything else must stay in lockstep. `tests/csp-parity.test.ts` (GH #408) fails fast on drift — both directives are diffed at test time so a future asymmetry trips CI before it ships.
- **Dev vs packaged in Electron is `app.isPackaged`, not `NODE_ENV`.** `NODE_ENV` isn't reliably set when Electron launches the main process; gate Electron-specific dev-only behaviour on `app.isPackaged ? prod : dev`.
- **The Windows ARM64 release job runs the full Vitest suite under x64 emulation, which makes `mongodb-memory-server`'s 10s startup timeout flaky** (`tests/compressImage.test.ts` was a victim on v1.25.0). When the job fails on that timeout specifically, a re-run usually passes; if it becomes a pattern, bump the `mongodb-memory-server` start timeout in `tests/setup.ts` or skip the affected test on the cross runner.
- **macOS installs from an older app cache the bundle ID, not the binary.** A `.dmg` in Downloads doesn't replace `/Applications/Filament DB.app` until you drag it over. Verify with `defaults read /Applications/Filament\ DB.app/Contents/Info.plist CFBundleShortVersionString`.
- **Windows auto-update channel is x64-only, by design (GH #586).** The Windows updater metadata file is `latest.yml` with NO arch suffix (unlike `latest-mac.yml` / `latest-linux-arm64.yml`). The x64 and arm64 Windows release jobs each emit one, so before #586 the later upload won — and an x64 user served the arm64 `latest.yml` auto-installed an arm64 build that can't run on x64, producing Windows' "Problem with Shortcut — 'Filament DB.exe' has been changed or moved" error intermittently (every other release, by CI upload-order luck). The fix: the `win-arm64-cross` job runs electron-builder with `--publish never` AND deletes `dist-electron/latest.yml` before the softprops upload, so **x64 is the single Windows update channel**. x64 installers run on arm64 Windows via the OS emulation layer, so arm64 users still auto-update (to the emulated x64 build); the native arm64 installer stays on the release for manual download. If native-arm64 auto-update is ever wanted, the proper fix is a post-build job that merges the two jobs' `latest.yml` into one multi-arch file — which is exactly what the mac side now does. **macOS auto-update is correct-arch as of #698 (v1.40.1).** The mac jobs had the same latent `latest-mac.yml` collision, now resolved: macOS builds are Developer ID-signed + notarized (v1.39.1), `mac.target` is `[dmg, zip]` (electron-updater can't auto-update from a dmg — it throws "ZIP file not provided"), and the `merge-mac-metadata` job combines both arches' single-arch `latest-mac.yml` into ONE multi-arch file. electron-updater's `MacUpdater` filters that file's `files` by the running arch (keeps `arm64`-in-name files on Apple Silicon/Rosetta, the rest on Intel), so arm64 Macs auto-update to arm64 and Intel to x64. Both mac builds pass `--publish never` (like `win-arm64-cross`) so `merge-mac-metadata` is the SOLE writer of `latest-mac.yml`. If that job is skipped/fails after the installers upload, the release ends up with NO `latest-mac.yml` — safer than a *wrong-arch* one, but NOT harmless: the packaged app still runs `checkForUpdates()`, which 404s on the missing metadata and emits `state: "error"` in `electron/auto-updater.ts` → the amber error banner. macOS auto-update is then broken (and noisy) until corrected. A *single-arch* `latest-mac.yml` essentially can't ship: the merge job only runs when both mac jobs succeed (`needs: build`), and `scripts/merge-mac-latest-yml.mjs` throws if either arch's entry is missing (unit-tested in `tests/mergeMacLatestYml.test.ts`). The residual failure is the merge job being skipped entirely (→ no `latest-mac.yml`, the 404 case above), which CI does NOT assert against — so after a release **manually confirm the asset exists and lists both `-mac-arm64.zip` and `-mac-x64.zip`**: `gh release download v<ver> -p latest-mac.yml` then check the `files` array. The `UpdateBanner` no longer gates the Download button off on macOS.

## PrusaSlicer Integration

- **Config bundle API**: `GET /api/filaments/prusaslicer` exports filaments as PrusaSlicer INI bundle; `POST` imports bundles back
- **Calibration API**: `GET /api/filaments/{id}/calibration?nozzle_diameter=0.4&high_flow=0|1` returns per-nozzle calibration data (extrusion multiplier, pressure advance, max volumetric speed, retraction); used by PrusaSlicer to auto-adjust filament settings when the user switches printer presets. Optional `high_flow` param disambiguates standard vs high-flow nozzles at the same diameter.
- **Spool check API**: `GET /api/filaments/{id}/spool-check?weight=42.5` checks whether any spool has enough remaining filament (by weight in grams) for a print job. PrusaSlicer calls this after slicing to warn if insufficient filament.
- **Sync with calibration context**: `POST /api/filaments/{id}?nozzle_diameter=0.4&high_flow=0|1` accepts optional query params so PrusaSlicer can write calibration-related keys (EM, PA, retraction) to the correct per-nozzle calibration entry
- **Match by stable id, not just name (#867, Phase 1)**: the sync historically keyed on the preset NAME, so a renamed preset stopped matching its filament and the fork's create-on-404 path silently spawned an orphan record that swallowed every later edit (a real "nothing syncs" footgun). The export (`filamentToSlicerKeys`) now round-trips `filamentdb_id` (the filament `_id`); the sync route (`POST /api/filaments/[id]`) resolves the target by **addressing mode** (Codex-hardened over 5 review rounds):
  - **ObjectId URL** (`POST /api/filaments/{ObjectId}`) is **authoritative** — resolve by that `_id`; a carried `filamentdb_id` must NOT override it (a copied/stale id in the config would otherwise redirect the write to the wrong row). A *found* ObjectId wins outright; if the 24-hex URL matches no `_id` (a preset legitimately NAMED with 24 hex chars), fall through to the name path below rather than 404. **Phase 2 companion (hyiger/PrusaSlicer#38):** the ObjectId path ALSO applies the request body's `name` to the record (a rename) when it differs — this is what makes the fork's "Update anyway" reconcile *stick* (otherwise the record keeps its old name and every later name-addressed sync re-hits `name_id_mismatch`). A rename colliding with another active filament returns **409 `name_taken`** (`conflictId` names it). The name-addressed path NEVER renames (the name is its addressing key; a `body.name` there is ignored).
  - **NAME URL** (the fork's normal sync): resolve **filamentdb_id → name**, falling back gracefully when the id is absent/stale (it's DB-instance-specific).
  - **Name/id mismatch ⇒ 409, no mutation**: when a name-addressed sync's `filamentdb_id` resolves to a filament whose stored name differs from the preset name, the case is ambiguous — a *renamed preset* (id right) vs a *copied/cloned id* (id points at the wrong filament, e.g. a Save-As that kept the source's id), indistinguishable server-side. So the route **does not apply the update** (a copied id would silently overwrite the source); it returns **409 `name_id_mismatch`** with `{filamentId, matchedName, sentName}` so the fork prompts. To update the resolved filament anyway, re-sync by its ObjectId URL (the authoritative form). This also prevents the orphan (no 404 → no create-on-404).
  - `filamentdb_id` is in `STRUCTURED_KEYS` so it's consumed for routing only, never stored in the settings bag. The **200** response (always = applied) carries `matchedBy` (`"id"|"name"|null`) + `matchedName` so the fork can re-stamp the id into a name-matched preset.
  - **Fork side (the `filamentdb_id` config option + send/clear-on-clone + the 409 reconcile prompt) is hyiger/PrusaSlicer#36; PrusaSlicer drops unknown config keys on load, so the key only persists with the FilamentDB fork.**
- **Field mapping**: `src/lib/prusaSlicerBundle.ts` maps structured DB fields → PrusaSlicer INI keys, merges with `settings` bag
- **Nil handling**: Structured DB fields that are null must NOT emit `nil` in the INI output — PrusaSlicer interprets nil as "reset to zero" for numeric fields. Only settings bag nil values (meaning "inherit from parent") are preserved.
- **compatible_printers default**: `filamentToSlicerKeys` emits `compatible_printers = ` and `compatible_printers_condition = ` (both empty) by default, which PrusaSlicer treats as "no restriction" — synced filaments show up in every printer's dropdown and the scan-stream auto-select works regardless of active printer. Without this default PrusaSlicer treats the filaments as "compatible with no printer" and filters them out of every active printer's filament list. Gated on the settings bag not already pinning a restriction, so a user-set `compatible_printers` from a round-trip import survives the next export (PR #235).
- **PrusaSlicer Filament Edition**: [hyiger/PrusaSlicer](https://github.com/hyiger/PrusaSlicer) has a `FilamentDB` module that fetches presets on startup via the REST API, syncs changes back with per-nozzle calibration context
- **NFC scan → preset select (SSE)**: `GET /api/scan/stream` is a long-lived Server-Sent Events endpoint that emits each NFC tag read (after the renderer matches it against the DB) as a `scan` event. Event payload: `{ timestamp, filament, candidates, decoded }` — `filament` is the matched DB row or null; PrusaSlicer keys presets by name so the consumer reads `filament.name` and calls its own "select filament for current extruder" path. The renderer pushes via `POST /api/scan/publish`; the in-process bus lives in `src/lib/scanBus.ts` (Node EventEmitter on `globalThis`). On connect the most recent scan is replayed once as a `replay` event so a slicer opened just after a tag read still picks it up; suppress with `?replay=0`. Heartbeats (`: hb`) every 25s keep proxies from idling the connection out. **In-process means one Filament DB instance, not one physical machine** — subscribers can be anywhere reachable over HTTP (a Pi running Filament DB can drive PrusaSlicer on a Mac across the LAN). What pins to a single machine is the publisher: NFC reads are emitted from the Electron renderer's `NfcProvider`, so the reader must be plugged into whichever box runs the Electron app. A headless Docker / web-only deploy has no `NfcProvider` and never publishes. A horizontally-scaled multi-process deployment would need an external broker behind the bus.
- **Port**: Dev and desktop use port **3456** (`next dev -p 3456`, Electron startup). Docker exposes the app on container port **3000** and is normally mapped with `-p 3456:3000`. PrusaSlicer defaults to `http://localhost:3456`.

## OrcaSlicer Integration

- **Config bundle API**: `GET /api/filaments/orcaslicer` exports filaments as an OrcaSlicer bundle; `POST /api/filaments/{id}/orcaslicer` syncs a single preset back. Mirrors the PrusaSlicer shape — the field mapping in `src/lib/orcaSlicerBundle.ts` differs in key names but the round-trip + nil-handling rules are the same.

## OpenTag3D read support (#864)

Read-only support for **OpenTag3D** tags (Polar Filament, American Filament, Numakers, 3D-Fuel, Ecogenesis), alongside the existing OpenPrintTag + Bambu readers — Filament DB as a neutral multi-standard reader. **Encoding settled empirically:** OpenTag3D v1.000 is a **FIXED BINARY MEMORY MAP** (absolute byte offsets, big-endian ints, UTF-8 strings, RGBA quads) — NOT CBOR. The "is it CBOR?" confusion came from conflating it with **OpenPrintTag** (Prusa, CBOR, MIME `application/vnd.openprinttag`), a *different* standard. OpenTag3D uses MIME `application/opentag3d`, and **both records can coexist on one physical tag**, so codec dispatch is per-NDEF-record-MIME, NOT per-chip.

- **Codec (`src/lib/opentag3d.ts`, DB-free)**: single source-of-truth field table transcribed from the authoritative `https://opentag3d.info/spec.json` (Core `0x00–0x65` with reserved gaps `0x0C–0x1A` + `0x4F`; Extended `0x70–0xBA`). Decoding rule: `real = raw * scaling`. `encodeOpenTag3D`/`decodeOpenTag3D` are symmetric over the table (round-trip tested). Tag-version policy: refuse a newer MAJOR, warn-but-parse a newer MINOR. `src/lib/opentag3d-decode.ts` maps decoded fields onto the shared `DecodedOpenPrintTag` shape (`tagSource: "opentag3d"`); OpenTag3D-only fields with no first-class home (MFI block, spool-core diameter, `td` mm, the vso triple, serial, online URL) ride `aux.opentag3d_*` so nothing is lost. **Unit traps:** `dry_time` is HOURS (converted ×60 to the minutes `dryingTime` expects); `td` is mm and is intentionally NOT mapped onto `transmissionDistance` (OpenPrintTag's HueForge TD — different quantity).
- **Pluggable codec registry (`src/lib/tagCodecs.ts`)**: `selectCodec`/`decodeFromNdefRecords` pick the first registered codec whose `ndefMime` matches a present TNF=0x02 record (OpenPrintTag preferred when both coexist). Shared by the desktop reader AND the server route — one decoder, no drift. Bambu MIFARE stays a separate non-NDEF path but emits the same decoded shape.
- **NDEF generalization (`src/lib/ndef.ts`)**: `parseNdefRecords(raw, ccOffset)` returns ALL records (`{tnf,type,payload}`); `ccOffset=0` for Type-5 (SLIX2), `12` for Type-2 (NTAG, CC at page 3 / byte 12, TLV at byte 16). `parseNdefFromTag` is now a back-compat wrapper (OPT payload only). `parseNdefRecordsAuto` tries `0` then `12`. Type-2 build helpers (`buildMediaNdefRecord`/`buildNdefMessageTlv`/`buildType2Cc`) feed the dev write CLI.
- **Desktop NTAG transport (`electron/nfc-service.ts`)**: `readNtagTag` reads NTAG via the `FF B0` READ BINARY pseudo-APDU (same style as the Bambu MIFARE path; no auth, no `FF FB` pass-through), validates the Type-2 CC, assembles the image, decodes via the registry. `readTag()` order: Bambu (MIFARE auth) → NTAG (`FF B0`; null = "not Type-2" → fall through, throw = real Type-2 content error → surface) → ISO-15693 (SLIX2; now registry-routed so an `application/opentag3d` record on SLIX2 decodes too). **Hardware-unverified** — implemented to spec + the Bambu `FF B0` precedent, not yet validated on an ACR1552U + NTAG.
- **Server auto-sniff (`/api/nfc/decode`)**: accepts `tagType: "opentag3d"`; a raw `tagMemory` dump is auto-sniffed (CC-offset + MIME) via the registry regardless of the `tagType` hint, so the mobile client needs no format-detection. `payload`-only (pre-parsed) uses `tagType` to pick the codec.
- **Dev write harness (`scripts/write-opentag3d-tag.ts`, `npm run nfc:write-opentag3d`)**: flashes a blank NTAG via `FF D6` UPDATE BINARY. Predates the product write path below; still handy for generating a test tag from CLI flags without a commercial spool, and the encoder doubles as the test-fixture source.
- **Product write/erase/read-only (tag-type-aware)**: the detail-page **Write NFC** button and **Settings → Devices → NFC Tools** (Erase, Set read-only / Make writable) auto-detect the loaded chip and act on the right standard — **SLIX2 → OpenPrintTag** (ISO-15693 `FF FB`), **NTAG213/215/216 → OpenTag3D** (`FF B0`/`FF D6` Type-2). `NfcService.detectTag()` (IPC `nfc-detect-tag`) reports `{family, standard, formatted, readOnly}`; the renderer picks the encoder (`filamentToOpenTag3DFields`→`encodeOpenTag3D` for NTAG via `src/lib/opentag3d-encode.ts`, vs `generateOpenPrintTagBinary` for SLIX2) and sends a `standard` discriminator on `nfc-write-tag`; the service routes the transport and refuses a chip/standard mismatch (`TAG_TYPE_MISMATCH`). The SLIX2 path is unchanged (extracted to `writeSlix2Impl`/`formatTagImpl`/`setReadOnlyImpl`); the NTAG path adds `writeNtagImpl`/`formatNtagImpl`/`setReadOnlyNtagImpl` + `writeNtagPage`.
  - **NTAG read-only is NOT supported** (read-only is a SLIX2/OpenPrintTag-only feature). Hardware testing showed the NTAG21x Capability Container (page 3) is **OTP** — a write that clears a CC bit returns SW 9000 but the bit persists (the read-only nibble goes `0x0→0xF` but never back). So a CC-nibble read-only is irreversible, and the static/dynamic lock bytes are likewise OTP/bricking — there's no reversible NTAG read-only. `setReadOnly` for an NTAG throws `NTAG_READONLY_UNSUPPORTED`; the NFC Tools read-only buttons disable when an NTAG is loaded. The write path does NOT honor the CC read-only nibble either (it's unreliable + can't be undone — a genuinely lock-byte-locked tag instead fails the page write naturally). **Writes only ever touch page 3 (CC), page 4 (TLV start), and page 5→last-user-page (zero-fill) — the static lock bytes (page 2) and dynamic/config lock pages are NEVER written.**
  - **Blank-NTAG sizing**: a blank tag has no CC to read its size from, so `getNtagNdefBytesViaGetVersion` issues `GET_VERSION` (`60h`) and maps the NXP storage-size byte (datasheet §8.3.7: `0x0F`=213/144B, `0x11`=215/496B, `0x13`=216/872B); an unrecognized size **refuses** rather than guessing (a wrong size could write past a smaller chip's end). A formatted tag's existing CC is trusted (the common case).
  - **Hardware status**: `FF D6` page write is proven on the ACR1552U+NTAG215 (the dev harness); the full product write/erase/read-only flow + `GET_VERSION` are **pending on-device confirmation**. NTAG *read* (incl. the read dialog + `aux.opentag3d_*`) is hardware-verified.
- **UI**: `NfcReadDialog` gains an OpenTag3D provenance badge + an extras list for the `aux.opentag3d_*` fields; the NFC Tools card shows the loaded tag type (e.g. "NTAG215 · OpenTag3D").
- **No upstream test vectors exist** (the canonical repo is a docs-only Jekyll site), so fixtures are hand-built from the spec's per-field `examples`. Tests: `tests/opentag3d.test.ts`, `tests/tagCodecs.test.ts`, plus route + ndef coverage.

## Multi-color filaments

Spec-aligned data model, end-to-end:

- **Schema** (`src/models/Filament.ts`): `color` stays nullable with the back-compat default `"#808080"`; `secondaryColors` validates max-5 + every entry matches `^#[0-9A-Fa-f]{6}$`. The `coextruded` form path explicitly sets `color = null` on submit so the spec-pure shape (null primary + populated secondaries) round-trips through Mongoose. Bambu MIFARE tags carry only one color so reading a Bambu tag populates the primary only.
- **Helpers** (`src/lib/filamentColors.ts`, DB-free for unit-test ergonomics): `deriveArrangement(optTags)` → `"solid" | "coextruded" | "gradient"` (priority `coextruded > gradient`); `displayColor(filament)` falls through `color || secondaryColors[0] || "#808080"` for UI surfaces that MUST show a single hex; `allColors(filament)` returns the full primary+secondaries list for swatch rendering; `isMultiColor(filament)` for gating notices and form sections.
- **Resolver** (`src/lib/resolveFilament.ts`): `secondaryColors` is array-fallback inheritable like `optTags` / `bedTypeTemps` — a variant either declares its own non-empty array or inherits the parent's whole array. Setting a variant's array to `[]` does NOT clear; it falls through to the parent. To override and show as single-color, give it at least one secondary slot or a different arrangement.
- **Read path** (`/api/filaments` list aggregation + `/api/filaments/[id]` variants subquery + parent summary): `secondaryColors` rides the projection, with effective-array merge mirroring `optTags` — same `$ifNull` + parent-`$lookup` pattern. Dashboard + filament list both consume `displayColor()` for their one-color spots.
- **`<FilamentSwatch>`** (`src/components/FilamentSwatch.tsx`): extended with `secondaryColors?: string[]` + `arrangement?: ColorArrangement`. Coextruded renders as hard-stop stripes via `linear-gradient(to right, c0 0%, c0 N%, c1 N%, ...)`; gradient renders as smooth `linear-gradient`. The finish overlay (matte / silk / sparkle / glow / translucent / transparent) still composes on top. Single-color callers are unchanged (backward-compatible). `FilamentForm`'s `MultiColorEditor` sub-component drives an arrangement radio (sets `optTags` 28/29 for coextruded — dual vs triple by color count — or 27 for gradient), 0–5 secondary color slot editor, and a live preview swatch that passes `null` color when coextruded for WYSIWYG.
- **OpenPrintTag encode/decode** (`src/lib/openprinttag.ts` + `-decode.ts`): `OPT_KEY.SECONDARY_COLOR_0..4` (CBOR keys 20–24). Encoder loops `secondaryColors.slice(0, 5)` into the right slots, parses each as RGBA, emits CBOR bytes. Decoder reads slots in order and surfaces them as `decoded.secondaryColors`. **Documented spec gap**: the alpha byte from `color_rgba` is truncated (`bytesToRgbHex` returns `#rrggbb` not `#rrggbbaa`) — translucency is modelled via finish optTags with real CSS opacity, not the alpha byte, which would be redundant. Round-trip nfc test pins this.
- **OPT DB import** (`src/lib/openprinttagBrowser.ts` + `/api/openprinttag/import`): `OPTMaterial.secondaryColors` parsed from raw `secondary_color_0..4` YAML fields. When a material has secondaries, the importer preserves a null primary on update (`color: m.color || (m.secondaryColors.length > 0 ? null : "#808080")`) AND clears the gray sentinel `"#808080"` to null on conditional-update when adopting coextruded secondaries — so re-importing an OPT material that's coextruded doesn't leave a stale gray primary.
- **NFC scan → form prefill** (`src/components/NfcReadDialog.tsx` + `/filaments/new/page.tsx`): `secondaryColors` ride as a comma-separated URL param, validated entry-by-entry as `#RRGGBB`, threaded through `initialData`.
- **CSV import/export** (`src/lib/importFilaments.ts` + `exportFilaments.ts`): `Secondary Colors` column (comma-separated). Importer parses, trims, drops malformed, caps at 5. When `secondaryColors` has entries AND `row.color` is null/empty, the importer explicitly sets `doc.color = null` to preserve a null primary on coextruded CSV round-trip.
- **Slicer-bound export — drop secondaries, never invent gray** (`src/lib/prusaSlicerBundle.ts` + `orcaSlicerBundle.ts`): a local `slicerExportColor(filament)` helper falls through `color → secondaryColors[0] → null` (returns `null` rather than `displayColor()`'s `"#808080"` sentinel so `set()` is a no-op when neither exists — the slicer uses its own default rather than getting forced to a gray the user never picked, the Codex P2 fix on PR #485). Single-filament + bulk exports both pass through this helper. Notice surfaces on the "Export for slicer" disclosure when `secondaryColors[].length > 0` so the user sees the trade-off before download.
- **OpenAPI** (`public/openapi.json`): `Filament.color` marked `nullable: true` with a description naming the OPT spec gap (alpha byte not stored), `Filament.secondaryColors` defined as a max-5 array of `^#[0-9A-Fa-f]{6}$` strings, both notes call out the slicer-export drop.

## Color-name typeahead → hex

`FilamentForm`'s "Color Name" input is a combobox: typing populates a dropdown of suggestions, picking one sets BOTH the color name and the hex color picker. Two suggestion sources, in priority order:

- **From your filaments** — the user's previously-saved `(colorName, color)` pairs, served by a new `GET /api/filaments/colors` endpoint that runs `$group` over non-deleted filaments. Multiple filaments with the same name+hex collapse to one row; different hexes under the same name (e.g. three brands of "Galaxy Black" with slightly different shades) stay separate so the user can pick the right shade.
- **Standard colors** — the 148 CSS Color Module Level 4 named colors plus the gray/grey + cyan/aqua + fuchsia/magenta + slategray/slategrey synonyms. `src/lib/cssNamedColors.ts` is the source of truth.

Lookup helpers in `src/lib/cssNamedColors.ts`:
- `lookupCssNamedColor(name)` — case- and whitespace-insensitive. Returns the canonical uppercase hex (`#000080` for any of `navy`, `Navy`, `  NAVY  `) or `null`.
- `filterColorSuggestions(dbSuggestions, query, maxResults?)` — substring-matches the query against both sources, dedupes by `(lowercase name, uppercase hex)` (DB wins ties), and orders DB entries above CSS entries. Exported because the combobox can't unit-test through it.

Commit-on-write rules in the combobox. **All hex-writing commits are gated on the `colorHexUserSet` flag (#794, v1.55):** a name commit fills/replaces the hex ONLY when the hex isn't "user-owned" — i.e. it's blank/incomplete, never picked, or was itself auto-filled by a prior name commit. Once the user edits the hex via the color picker or hex text field it's owned, and from then on a name is just a *label* and never overwrites it.
- **Click / Enter on a dropdown row** sets `colorName`, and sets `color` only when the hex isn't user-owned.
- **Tab-out / blur with an exact CSS named-color match** updates the hex (even without opening the dropdown) only when the hex isn't user-owned — covers users who already know "navy" and just want to type it.
- **DB-side blur lookup is intentionally NOT auto-applied** — the same name can have multiple legitimate hexes in the DB; the user has to pick which one they meant via the dropdown.
- **Plain typing never overwrites the hex** — only an explicit, non-user-owned commit (click/Enter/exact-CSS-on-blur) changes the color picker, so the user's free-text "I'm naming this whatever I want" path stays uninterrupted.

Translation keys: `form.colorName.section.db` ("From your filaments" / "Aus deinen Filamenten") and `form.colorName.section.css` ("Standard colors" / "Standardfarben") in both `en.json` and `de.json`.

## Finish indicator on the swatch + name chip

When two filaments share the same color but a different finish (the canonical case: white plain / matte / silk), the inventory list, detail page header, and the color-variants list under a parent now distinguish them visually:

- **`<FilamentSwatch>` finish texture**: a new `finish?: "matte" | "silk" | "sparkle" | "glow" | "translucent" | "transparent" | null` prop drives a CSS overlay on top of the color. Matte = flat fill (no highlight), silk = soft sheen gradient, sparkle = four small dots whose color flips based on the underlying fill's luminance, glow = soft yellow inner halo, translucent/transparent = real alpha (55% / 25%) over a checkered backdrop (the universal "see-through" signal). All five non-alpha treatments compose with `data-finish="..."` for QA snapshots.
- **`<FinishChip>`** renders the same finish as a small uppercase chip beside the filament name. Per-finish color palette (silk = blue, sparkle = amber, glow = yellow, translucent = cyan, transparent = slate, matte = gray) and reuses the existing chip styling vocabulary.
- **Source of truth**: `src/lib/filamentFinish.ts`'s `deriveFinish(optTags)` maps optTag IDs to a single canonical finish string. Priority order when a filament carries multiple finish-relevant tags: `transparent > translucent > sparkle > silk > glow > matte`. Parents are finish-agnostic — finishes are only derived for variants and standalones; `<FilamentSwatch isParent>` short-circuits to the parent render (composite-of-variant-colors since GH #597, cross-hatch fallback) regardless of finish.
- **Data plumbing**: `optTags` rides the list aggregation (`GET /api/filaments`) and the per-parent variant projection in `GET /api/filaments/{id}` (`.select("name color cost optTags")`) so neither the list page nor the detail-page color-variants list need a follow-up fetch. `FilamentSummary` and `FilamentVariant` types gain `optTags?: number[]`.
- **Translations**: new keys `swatch.finish.matte`/`silk`/`sparkle`/`glow`/`translucent`/`transparent` in both `en.json` and `de.json`. Kept distinct from the existing `form.tag.*` keys so chip text can diverge from form-checkbox labels without coupling them.

## Trusted-origin (CSRF) guard sweep

`assertSameOriginRequest` (`src/lib/requestGuard.ts`) was originally added in GH #252 for snapshot export·restore·wipe + the MongoDB connection test + the Atlas import (the obviously destructive admin handlers). GH #360 extended it to every other mutating API route — POST/PUT/DELETE/PATCH on filaments, nozzles, printers, locations, bed-types, print-history, spools and assignments, share + share/[slug], scan/publish, prusament/import, openprinttag/import, tds, and all the import-* endpoints. Same pattern at the top of each handler:

```ts
const guard = assertSameOriginRequest(request);
if (guard) return guard;
```

The guard rejects browser cross-origin (`Sec-Fetch-Site: cross-site` / `same-site`, or an `Origin` whose authority doesn't match `Host`) with 403, and lets non-browser clients (curl, PrusaSlicer / OrcaSlicer integrations) pass — they send neither header. **One stub left unguarded on purpose**: `POST /api/filaments/orcaslicer` is a 501 placeholder for the not-yet-implemented bulk OrcaSlicer importer; no destructive op runs and the handler takes no `request` argument. When bulk import lands, add the guard then.

Regression coverage in `tests/destructive-route-guard.test.ts` pins a representative sample (filaments / nozzles / locations / share / scan-publish / spools-import / print-history) so a future contributor can't quietly drop the guard from a new route. The Codex audit issue #360 has the full inventory.

## Brother PT-P710BT label printer

End-to-end USB label printing for the Brother PT-P710BT (P-touch CUBE) — Electron-only feature, shipped in v1.34.0 (PR #487), transport reworked from Bluetooth-serial to the OS print system in GH #588 (v1.34.x) after a hardware test found the original path couldn't reach the USB device. Architecture worth knowing about before touching the code:

- **Transport is the OS print system, not serialport (GH #588).** On the desktop the PT-P710BT's Bluetooth is iOS/Android only (Brother's official line); over USB it's a **USB printer-class** device, NOT a serial port — so `serialport` (the old transport) could only ever reach the unsupported, flaky Bluetooth-SPP node, never the USB device. The current transport hands the raster byte stream to the platform print stack, which owns the USB device + driver: **macOS/Linux → CUPS** (`lp -o raw`; for a raw `usb://…` device with no installed queue we auto-manage a hidden raw queue `FilamentDB_Label` bound to it via `lpadmin`); **Windows → the spooler** (`RAW` datatype via a PowerShell winspool `WritePrinter` P/Invoke). `serialport` and its native-binding loaders (`@serialport/*`, `node-gyp-build`, `node-addon-api`) were dropped entirely. macOS USB is hardware-validated; Windows + Linux are implemented but need on-hardware verification.
- **Encoder is pure JS, browser-safe** (`src/lib/labelEncoder.ts`). Implements Brother's [Raster Command Reference for PT-E550W/P750W/P710BT](https://download.brother.com/welcome/docp100064/cv_pte550wp750wp710bt_eng_raster_102.pdf). Shared by the renderer dialog, the standalone CLI (`scripts/print-label.ts`), and the simulator that decodes the byte stream back into a PNG (`scripts/print-label-sim.ts`). Tape width 24mm rounds to media-byte 24; 3.5mm rounds to 4 (Brother's wire convention — `Math.round` covers both).
- **Bitmap rendering** (`src/lib/labelBitmap.ts`) is browser-only — uses HTMLCanvasElement + the `qrcode` npm. Renders QR + filament name at the print head's native 128-dot density, rotates 90° for raster-line orientation, thresholds to 1-bit grayscale ready for `packGrayscaleBitmap()`. **Orientation fix (GH #587):** after the 90° rotation it **reverses the raster-line order** — feeding lines in the rotate-order printed the label MIRRORED along its length (text backwards, QR reversed), confirmed on a real PT-P710BT. The printer's physical feed direction is opposite the raster-line order; reversing it un-mirrors the label. The CLI (`scripts/print-label.ts`) carries the identical reversal. QR sizing picks the largest module scale that fits `PRINT_HEAD_DOTS - 2 * VERTICAL_PADDING_DOTS = 116` dots after including the spec-required 4-module quiet zone (rendered with `margin: 4`). Payloads too long to fit even at scale 1 throw — no silent clipping.
- **Transport** (`electron/label-printer.ts`, GH #588) shells out to the OS print system — no native deps. `listLabelPrinters({ probeUsb })`: macOS/Linux parse `lpstat -v` (installed queues) and — **only when `probeUsb` is true** — `lpinfo -v` (available `usb://` devices), surfacing the managed queue as its underlying device URI and deduping; Windows runs `Get-Printer` via PowerShell. **GH #771: `probeUsb` defaults to false.** On macOS `lpinfo` issues the admin-only CUPS `CUPS-Get-Devices` op, which pops the OS authorization (admin password) dialog — so the Settings panel lists passively (installed queues, no prompt) on mount and only probes USB on an explicit Refresh, where a credential prompt is expected. `lpstat` is a plain read and never prompts. `printLabel(target, bytes)`: a `usb://…` target routes through `ensureManagedQueue()` (idempotent `lpadmin` create/rebind, no `-m` → raw queue) then `lp -d <queue> -o raw` with bytes piped to stdin; an installed-queue-name target prints directly; Windows writes a temp file + drives a winspool `WritePrinter` RAW P/Invoke. Per-subprocess 15s timeout (`SIGKILL` on stall); the IPC handler wraps the whole print in its own 30s timeout. `tests/label-printer.test.ts` mocks `node:child_process` and pins: listing parse + managed-queue/lpinfo dedup, direct-queue print (no lpadmin), managed-queue create-then-print for a `usb://` device, idempotent skip when already bound, lp non-zero-exit → reject with stderr, and spawn error → reject. The Windows branch (`process.platform === "win32"`) isn't covered by CI — needs a real Windows host.
- **IPC handlers** (`electron/main.ts`, all `assertTrustedSender`-gated): `label-printer-list-devices`, `label-printer-get/set-device-path`, `label-printer-get/set-public-url`, `label-printer-print`. The Uint8Array byte stream rides through IPC as a plain `number[]` to sidestep cross-version `structuredClone` quirks; renderer-side `Array.from(bytes)` is O(n) but cheap at the ~270 KB max-label size.
- **Public-URL setting** (`labelPrinterPublicUrl` in electron-store) is required for URL-mode QR codes in packaged Electron — `window.location.origin` is `http://localhost:3456` which scanners on other devices can't reach. Validated server-side in `electron/main.ts`: must be `http(s)`, must not be a loopback host (via the shared `isLoopbackHostname` in `src/lib/loopbackHost.ts`), must have no query string or fragment (raw-input check — bare `?` / `#` slip past `parsed.search`/`parsed.hash`). Renderer-side `isLoopbackUrl(window.location.origin)` shares the same helper to disable URL mode when the dialog detects a localhost origin and no public URL configured.
- **`isLoopbackHostname`** (`src/lib/loopbackHost.ts`) is the single source of truth for "this hostname addresses the local machine". Covers every URL.hostname output we've seen: `localhost` (case-insensitive), DNS absolute-name notation (`localhost.`, trailing dots), 127/8, bare + bracketed IPv6 loopback (`::1`, `[::1]`, uncompressed `0:0:0:0:0:0:0:1`), IPv4-mapped IPv6 loopback in dotted + Node's hex-normalised form (`::ffff:127.x.x.x` and `::ffff:7fNN:YYYY`), and the unspecified-bind addresses (`0.0.0.0`, `::`, `[::]`, IPv4-mapped `::ffff:0.0.0.0` / `::ffff:0:0`). 56-case test suite in `tests/loopbackHost.test.ts` pins each shape end-to-end through `new URL().hostname`. **PR #487 went 12 Codex iteration rounds, mostly on this validator.**
- **Packaged build**: the label printer no longer bundles any native module (GH #588 removed `serialport`/`@serialport/*`/`node-gyp-build`/`node-addon-api` from `electron-builder.yml` and `package.json`) — it shells out to the OS print tools (`lp`/`lpstat`/`lpinfo`/`lpadmin` / `powershell`) which are always present. `npm run electron:rebuild` now rebuilds only `@pokusew/pcsclite` (the NFC native module): `-o @pokusew/pcsclite`. **Linux note:** creating the managed raw queue needs `lpadmin` rights; if it fails the transport surfaces a message telling the user to add the printer in their system print settings first (then it shows up as an installed queue and prints with no `lpadmin`).

Persisted by `src/hooks/useLabelFormat.ts` (electron-store on desktop / localStorage on web, mirroring `useCurrency`; JSON under key `labelFormat`/`filamentdb-label-format`). The renderer `renderLabelBitmap`/`renderLabelPreviewDataUrl` take `(filament, qrPayload, format)` and share one `composeLabelCanvas` so preview ≡ print. **Invert keeps the QR dark-on-light** (an inverted QR won't scan) by drawing it on its own white tile. Edited in `src/components/LabelFormatEditor.tsx` with a live preview over a built-in `SAMPLE_FILAMENT`; the editor is NOT Electron-gated (the format applies to the web `.bin` path too). The per-print QR *payload* mode (instanceId vs URL) stays in the dialog. `DEFAULT_LABEL_FORMAT` reproduces the pre-#592 output (QR left + name). No "remaining" field — a printed amount goes stale; scan the QR for the live value.

Case-insensitive fallback only when unique (a `ABC`/`abc` legacy collision returns both as `candidates` instead of an arbitrary pick). The schema doesn't enforce hex on `instanceId`, so legacy/imported IDs like `custom-id-123` still resolve. The 128-char cap is **route-level** (see `boundedParam` in `src/app/api/filaments/match/route.ts` after GH #513) — applied to name/vendor/type/instanceId alike to bound regex compile cost on the cross-origin-reachable endpoint. `escapeRegex` makes the case-insensitive comparison literal so regex-special chars in stored IDs (or in the query) don't cause false matches.
- **Wire-format byte traps** (avoid re-litigating these — pinned in `tests/labelEncoder.test.ts`):
  - `ESC i z` media width: 3.5mm tape rounds to byte 4 (Brother's wire convention); 6/9/12/18/24 round to themselves. `Math.round` covers both.
  - `ESC i M` mode bits: `0x40` enables auto-cut; `0x00` disables.
  - `ESC i K` various-mode flags: bit 3 (`0x08`) = no-chain (feed + cut after the LAST label). With `0x00` (chain mode) the autoCut bit still issues a cut command, but the tape isn't fed through the cutter and the label stays stuck in the head. The encoder ties this to the `autoCut` option: `autoCut: true → 0x08`, `autoCut: false → 0x00`. Don't hard-code either value.
  - Trailer: `0x1A` (print with feed/cut) when `autoCut: true`; `0x0C` (print, no feed) when `autoCut: false`.

**Caveats / known constraints** (post-#588 USB transport):
- **Windows + Linux are implemented but not yet hardware-tested** — only macOS USB is validated end-to-end. The Windows winspool `WritePrinter` RAW P/Invoke and the Linux `lpadmin` managed-queue path both need on-hardware confirmation.
- **The app auto-manages a hidden CUPS queue** (`FilamentDB_Label`) for raw `usb://` devices that aren't an installed queue — it appears in the user's printer list. Creating it needs `lpadmin` rights (fine for admin users on macOS; on Linux, if `lpadmin` fails the transport tells the user to add the printer in their system print settings first, then it shows up as an installed queue and prints with no `lpadmin`).
- **`lpinfo` is scoped to `--include-schemes usb`** — a bare `lpinfo -v` also runs the snmp/dnssd network backends, which probe the LAN ~13s and blow the 15s list-devices IPC timeout (#590).
- **Upgrades from the old Bluetooth/serial setting** (`/dev/tty.*`, `/dev/cu.*`, `/dev/rfcomm*`, `COMn`) are detected and rejected with a "select your printer again" message rather than failing obscurely on `lp -d /dev/...` (a CUPS queue name can't contain `/`).

## v1.48–v1.50: per-spool instanceId (#732)

The 5-byte hex `instanceId` identity moved from the FILAMENT to the SPOOL, so each physical roll has its own durable identity (the value NFC tags carry, QR labels encode, the slicer SSE bus keys on, and inventory CSV round-trips). The filament-level `instanceId` remains as a transitional fallback. Shipped in five phases across v1.48.0–v1.50.0.

- **Schema + migration — Phase 1 (#733, v1.48)**: `ISpool.instanceId: string` on the spool subdocument with `default: generateInstanceId` (`crypto.randomBytes(5).toString("hex")`, 10 hex chars) — `src/models/Filament.ts`. A pre-save hook stamps it; `FilamentSchema.index({ "spools.instanceId": 1 })` indexes it; `backfillSpoolInstanceIds()` migrates existing spools (the first id-less spool of a filament adopts the filament's top-level id — the "carry-over" — the rest get fresh ids). The dbConnect migration block (`src/lib/mongodb.ts`) runs the spool backfill (flag `spoolInstanceIds`) before the legacy filament backfill. `isSpoolInstanceIdTaken(instanceId, excludeSpoolId?, ownFilamentId?)` checks collisions against BOTH other spools' ids AND other filaments' top-level ids (the matcher resolves spool ids first, so a spool id equal to another filament's id would shadow it).
- **Resolve & write by spool — Phases 2 & 3 (#737, v1.48)**: `src/lib/matchFilament.ts` resolves a scanned id against `spools[].instanceId` FIRST (exact-case → CI), returning the matched spool in a new `matchedSpool` field, before the filament-level fallback. `POST /api/nfc/decode` propagates `matchedSpool` + `matchedBy` ("instanceId" | "heuristic"). `src/lib/selectSpoolForWrite.ts` (pure) picks WHICH spool's id a writer encodes: requested spool → first non-retired → first of any → filament-level fallback. The NFC write paths + `GET /api/filaments/{id}/openprinttag?spool=<id>` encode the selected spool's id (and its weight); the desktop scan bus carries the spool uid through to `ScanEvent`.
- **Edit / regenerate — Phase 4 (#739, v1.49)**: `validateSpoolInstanceId` (`src/lib/validateSpoolBody.ts`) — trim, 1–128 chars, `^[A-Za-z0-9._-]+$` (covers numeric Prusament roll ids, auto-gen hex, custom ids). `PUT /api/filaments/{id}/spools/{spoolId}` accepts a validated `instanceId` (409 on duplicate, 400 on bad charset) or `{ regenerate: true }` (mints a fresh id); `POST …/spools` accepts an optional explicit id. The SpoolCard exposes inline ID edit + a confirm-gated regenerate. The id is surfaced read-only on `/inventory` and the main list's spool panel (`by-location` aggregation projects it).
- **Export / import / share round-trip — Phase 5 (#742, v1.50)**: `src/lib/exportSpools.ts` emits each spool's OWN `instanceId` (was emitting the filament-level id for every row). `POST /api/spools/import` honors an `instanceId` column **on CREATE only** (validated + uniqueness-checked against other spools, other filaments' top-level ids, and other rows in the same CSV via an in-loop `Set`; auto-generated when absent); on the UPDATE path (a row whose `spoolId` matches an existing spool) the column is informational and the spool keeps its id — this makes a pre-Phase-5 export (which wrote the filament id into every row) round-trip idempotently. Snapshot export/restore preserves spool ids verbatim (filaments serialize whole). Shared catalogs strip `spools` + the top-level `instanceId`, so a per-spool id never leaks into a public share.

## v1.52: create/link variants from OpenPrintTag (#753)

Connects the OpenPrintTag import path to the variant/inheritance model so a variant can be created from — and stay linked to — an OPT material without the old "import standalone → set parent → hand-delete every inherited field" dance. Two of the issue's three approaches shipped: **C** (link an existing filament) as the foundation and **A** (parent-aware import) on top; the imperative "dedupe on re-parent" **B** was skipped as the only data-mutation-risky one. Builds on the v1.36 re-sync provenance system ([[#607]]).

- **Approach C — link an existing filament (`POST /api/filaments/{id}/openprinttag/link`)**: CSRF-guarded; body `{slug}`. Writes ONLY the linkage (`settings.openprinttag_slug`/`_uuid`) + the `openprinttagSnapshot` provenance via the shared `buildOptLinkUpdate(payload)` (`src/lib/optResync.ts`) — never a field value. So linking can't clobber a user-set or (for a variant) an inherited value: the existing `check` route diffs the variant's EFFECTIVE values, so a field equal to OPT's offer isn't surfaced and a diverged one classifies as `conflict` (user decides). Re-filters `_deletedAt: null` on the write (soft-delete race, mirrors `sync`). 404 `{linked:false, found:false}` when the slug is gone upstream. The bulk OPT import route was DRY'd to use `buildOptLinkUpdate` for its own link write.
- **Approach C UI**: a teal "Link to OpenPrintTag" button on the detail-page action bar, gated on `!filament._hasOwnOptLink` (the inverse of the v1.36 "Check for updates" gate). Opens `src/components/OptLinkDialog.tsx` — a focus-trapped material picker that fetches `GET /api/openprinttag` and renders results only once the user types (capped at 50 — no virtualization needed) → POST link → page refetch (the teal "Check for updates" button then appears).
- **Approach A — import AS A VARIANT (`POST /api/openprinttag/import` gains optional `parentId`)**: imports exactly ONE slug as a variant of the parent (single-slug enforced; parent must exist, be active, and not itself be a variant — `mongoose.isValidObjectId` guard → 400). Pulls only the DISTINCT fields onto the variant via `pruneOptPayloadAgainstParent(payload, parentEffective)` (pure, in `optResync.ts`): every inheritable field whose mapped value EXACTLY equals the parent's effective value is dropped/emptied so the variant inherits it dynamically (scalars/temps → null, the whole-array fields `optTags`/`secondaryColors` → `[]`). Never pruned: `color`/`colorName`/`name` (variant-only) and `vendor`/`type` (required identity — `PRUNE_SKIP_FIELDS`). `diameter` is force-nulled (`mapToFilamentPayload` hardcodes a fake 1.75). The snapshot is built from the FULL offer BEFORE pruning, so a pruned-then-edited field still classifies correctly on a later check. Create-only: a name collision with an existing active row is refused with 409 — it never re-parents/mutates another filament; the unique-name E11000 race recovers to the same 409. Returns the created `filament` in the response.
- **Approach A UI**: on the OPT browser (`src/app/openprinttag/page.tsx`), when exactly one material is selected, an "Import as variant of…" `<select>` (lazily-fetched root filaments — the non-variant rows of `GET /api/filaments`, fetched on first narrow-to-one) switches the import button to variant mode and threads `parentId`.
- **Known limitation (documented + tested)**: when the OPT material has EMPTY `secondaryColors`/`optTags` but the chosen parent has non-empty ones, the variant's `[]` inherits the parent's array (`resolveFilament` reads empty === inherit, GH #106/#477) — e.g. a single-color/no-tag material under a multi-color/tagged parent resolves to the parent's colors/tags. This is the SAME empty=inherit constraint already documented for resync in `OPT_CLEARABLE_ARRAY_FIELDS`; no stored array value resolves to empty while the parent is non-empty, so it can't be fixed without a model-level "no-inherit" sentinel (out of scope). The user can override by editing the variant after import.
- **Tests + docs**: `tests/optResync.test.ts` (pure `pruneOptPayloadAgainstParent` + `buildOptLinkUpdate` cases) and `tests/opt-resync-route.test.ts` (link route + variant import: variant-aware classification, strict-equality prune, collision/parent-validation/malformed-id guards, CSRF). OpenAPI documents the `link` endpoint + the `parentId` import param. New `optLink.*` + `openprinttag.importAsVariant`/`variantParent*` i18n keys (EN/DE in lockstep).

## Mobile scanner app (packages/mobile)

An Expo / React Native companion scanner — a deliberately **thin client** that does no business logic: it forwards scans + edits to the desktop/web server's REST API and renders the responses (field mapping lives server-side).

- **Layout**: `packages/mobile/` is a standalone in-repo package (NOT an npm workspace — the repo root has no `workspaces` field, so its `node_modules` is fully isolated from the web/Electron tree). Expo Router under `src/app/`: `index`, `scan-qr`, `create-from-tag`, `settings`, `filament/[id]`; helpers under `src/lib/`: `api`, `serverConfig`, `nfc`, `zeroconf`, `writeQueue`, `pendingScan`, `features`, `types`, `base64`, `theme`. The original plan lives in `docs/mobile-app-plan.md`. **Always run Expo from `packages/mobile/` only** (see the iOS-dev-build memory note).
- **Scanning**: QR/label scanning (`expo-camera`) resolves an `instanceId` via `GET /api/filaments/match?instanceId=…`, or a spool deep link's `?spool=<id>` via `GET /api/spools/{spoolId}`. NFC (`react-native-nfc-manager`) reads OpenPrintTag (NFC-V/ISO 15693) tags and POSTs the raw bytes to `POST /api/nfc/decode` for **server-side decode** — the phone does no CBOR/NDEF parsing. (Bambu MIFARE Classic is Android-only — iPhones can't read it.)
- **NFC feature gate**: `EXPO_PUBLIC_ENABLE_NFC` (mirrored in `src/lib/features.ts` as `NFC_ENABLED`, default on; `=0` produces a QR-only build for a free Apple ID that can't get the Core NFC entitlement — the NFC UI entry points hide so the build looks intentional, not broken).
- **Server config** (`src/lib/serverConfig.tsx`): the base URL + optional API key persist in `expo-secure-store`. The address is set manually OR auto-discovered via mDNS (`src/lib/zeroconf.ts`, `react-native-zeroconf` — gracefully no-ops with `supported: false` on web / Expo Go). When the server runs `FILAMENTDB_API_KEY`, the client sends `Authorization: Bearer <key>` on every request (harmless when the server is unauthenticated).
- **Create-from-tag** (`src/app/create-from-tag.tsx` + `api.createFromTag`): POSTs `{ tagData, overrides, spoolRemainingGrams }` to `POST /api/filaments`; the server maps the decoded tag → filament payload, the user's confirmed name/vendor/type override, and `spoolRemainingGrams` (null = catalog-only, no spool).
- **Per-spool updates**: move location, set remaining weight, retire/un-retire (`api.updateSpool` → `PUT …/spools/{spoolId}`), plus usage logging (`api.logUsage`) and dry-cycle logging (`api.logDryCycle`).
- **Offline write queue** (`src/lib/writeQueue.ts`): persists mutations in `AsyncStorage` and replays them FIFO when the server is reachable again — survives app restart, capped at 200 (oldest dropped first). **Only idempotent absolute-SET ops are queued** (`updateSpool` — weight/location/retire); `logUsage`/`logDryCycle` DECREMENT/APPEND so replaying a committed-but-lost response would double-apply — they require live connectivity and are never queued. A 4xx/5xx (real rejection) is surfaced/dropped, not queued; only network failures (`ApiError` status 0) queue.
- **Build & distribution (EAS, NOT Xcode Cloud)**: managed Expo / Continuous Native Generation — `ios/` and `android/` are generated by `expo prebuild` and **never committed** (gitignored), so Xcode Cloud has no Xcode project to build. Builds go through **EAS Build** instead (`packages/mobile/eas.json` — `preview` / `preview-qr` / `production` profiles; `preview-qr` sets `EXPO_PUBLIC_ENABLE_NFC=0` for a no-NFC, free-Apple-ID-signable build; `production` is `appVersionSource: remote` + `autoIncrement`). A repo-root `.easignore` scopes the upload to `packages/mobile` (EAS archives the git root by default), restating the secret/local excludes since `.easignore` replaces `.gitignore` for EAS. The `eas-build-mobile.yml` workflow (manual, needs `EXPO_TOKEN`) queues a cloud build; one-time `eas login`/`eas init` + an interactive iOS build to set up signing (the App ID needs the NFC Tag Reading capability + a paid account for an NFC build) are owner steps. Local dev stays on `expo run:ios`.

Note: PC/SC's `SCARD_STATE_PRESENT` bit can fire spuriously (driver-dependent — observed on ACS readers, especially the ACR1552U on macOS). The reader-status handler in `electron/nfc-service.ts` skips the FIRST event per reader to dodge the documented plug-in phantom (GH #230), but a later event with `SCARD_STATE_CHANGED` toggled can still set `tagPresent: true` without a real tag in the field. The recovery path: the auto-read in `electron/main.ts` fires on the present→true transition; if `connect()` exhausts its retries with `"Cannot connect to tag"`, we treat the present bit as phantom and call `NfcService.clearPhantomPresence()` to reset `tagPresent: false` in the service state. `clearPhantomPresence` ONLY touches the service-level `tagPresent` flag — it does NOT wipe the per-reader `readerPresent` map, because in multi-reader setups another reader may legitimately have a real tag whose presence event we already recorded. Without the corrective `tagPresent` clear, the renderer pill stays at "Tag detected" indefinitely and survives unplug+replug. The same first-event skip also drops a tag *already resting* on the reader at connect time (it produces no later state-change event); #572 handles that by emitting `presentAtConnect` on the skipped present event for a one-shot silent verification read — see the "v1.34.8–1.34.10 Features" section.
